# セッション管理

## 概要

KOTONOHAはRunPod Serverlessを「セッション単位」で使用する。ワーカーはリクエスト単位で使い捨てではなく、セッション中は同じワーカーが生き続ける。

## ワーカーIDによるセッション維持

1. 最初のリクエスト（runsync/run）でワーカーが起動する
2. レスポンスにワーカーIDが含まれる
3. 以降のリクエストでワーカーIDを指定し、同じインスタンスにルーティング
4. リクエストが途絶えるとアイドルタイムアウトでワーカーが自動停止

## セッション開始フロー

```
クライアント                          RunPod
    |                                   |
    |-- runsync {pubkey: "ssh-..."}---->|
    |                                   |-- ワーカー起動（コールドスタート）
    |                                   |-- entrypoint.sh実行
    |                                   |-- Ollama起動、モデルロード
    |                                   |-- handler.py: pubkey設定、sshd起動
    |<-- {status: ok, worker_id: xxx} --|
    |                                   |
    |-- runsync {messages: [...]} ----->| (worker_id指定)
    |<-- {response: "..."} ------------|
    |                                   |
    |-- SSH接続 ---------------------->| (同じワーカーのIPに接続)
    |                                   |
    |   ... (アイドル) ...              |
    |                                   |-- タイムアウト → ワーカー停止
```

## チャット履歴

- ワーカー側に会話状態を持たない
- クライアントが全履歴を`messages`配列で毎回送信
- ワーカーが停止しても会話履歴はクライアント側に残る
- 新しいワーカーが起動しても、同じ履歴を送れば会話を継続できる

## SSH接続

- pubkeyはAPIリクエストでワーカーに設定する
- 環境変数`PUBLIC_KEY`による起動時注入も可能（RunPodテンプレート設定）
- 1ワーカー = 1ユーザーの鍵のみ（上書き方式）
- ワーカー停止で鍵は消える

## タイムアウト設定

RunPod Serverlessのアイドルタイムアウトで制御する。

- **Execution Timeout**: 1リクエストの最大実行時間
- **Idle Timeout**: リクエストがない場合にワーカーを停止するまでの時間

セッション維持のためにはIdle Timeoutを適切に設定する必要がある。短すぎるとチャット中にワーカーが停止し、長すぎるとコストが増加する。

## クライアント実装の責務

- ワーカーIDの保持
- チャット履歴の保持と送信
- ワーカー停止の検知と再起動ハンドリング
- SSH接続情報（IP、ポート）の管理
