# アーキテクチャ

## 概要

KOTONOHAはRunPod Serverlessを「セッション単位」で利用する。ユーザーがリクエストを投げるとワーカーが起動し、同じワーカーに対してチャットやSSH接続を行う。リクエストが途絶えるとアイドルタイムアウトでワーカーが自動停止する。

## CIELとの違い

| | CIEL（画像生成） | KOTONOHA（LLMチャット） |
|---|---|---|
| ワーカー | 共有型（複数ユーザー） | 専有型（1ユーザー） |
| 料金 | 使用秒数で按分 | 全額負担 |
| 状態 | ステートレス（画像生成して終わり） | セッション単位（ワーカーが生き続ける） |
| SSH | なし | あり |

## セッションライフサイクル

```
1. ユーザーがリクエストを送信（runsync/run）
2. RunPodがワーカーを起動（コールドスタート）
3. entrypoint.sh: Ollama起動 → モデルロード → sshd起動 → handler.py起動
4. handler.pyがリクエストを処理（チャット or pubkey設定）
5. レスポンスにワーカーIDが含まれる
6. 以降のリクエストはワーカーIDで同じインスタンスにルーティング
7. リクエストが途絶える → アイドルタイムアウト → ワーカー自動停止
```

## コンポーネント構成

```
[クライアント（test_chat.py / PHP / ブラウザJS）]
  ├── RunPod API にリクエスト送信
  ├── ワーカーIDを保持してセッション維持
  ├── チャット履歴をクライアント側で保持
  └── SSH接続（公開鍵設定後）

[RunPod Serverless ワーカー]
  ├── Ollama（LLM-jp-4）  ← LLM推論
  ├── handler.py            ← RunPodリクエスト処理（チャット / pubkey設定）
  └── sshd                  ← SSH接続受付（pubkey設定後に起動）
```

## ステートレス設計

- ワーカー側に会話状態を持たない
- チャット履歴はクライアントが保持し、毎回全メッセージを`messages`配列で送信
- ワーカーが停止すればデータは全て消える（使い捨て）
- SSH公開鍵もワーカー停止で消える
